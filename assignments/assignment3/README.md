# 3rd Assignment: Project Proposal

## 과제 개요

이번 과제부터 Assignment 6까지 진행될 **ML 기반 도구/서비스/어플리케이션 만들기 프로젝트**의 첫 단계입니다.
이 프로젝트를 통해 Machine Learning을 활용한 실용적인 도구를 직접 기획하고 구현하게 됩니다.

**이것은 개인 프로젝트입니다.** 각자 본인의 문제를 해결하는 도구를 만들어보세요.

### 전체 프로젝트 구성

| Assignment | 주제 | 설명 |
|-----------|------|------|
| **Assignment 3** | **Project Proposal** | 프로젝트 주제 설정 및 제안서 작성 |
| Assignment 4 | Data Collection and Analysis | 데이터 수집 및 분석 |
| Assignment 5 | Model Training and Evaluation | 모델 학습 및 평가 |
| Assignment 6 | Real Usage and Final Report | 실제 사용 및 최종 보고서 |

### Assignment 3의 목표

프로젝트 아이디어를 구체화하고 제안서를 작성합니다. **가장 중요한 것은 본인이 실제로 5번 이상 사용할 만한, 본인의 실제 문제를 해결하는 도구여야 한다는 것입니다.** 어떤 문제를 해결할 것인지, 어떤 데이터와 ML 기법을 활용할 것인지 정리하는 단계입니다.

### 중요한 요구사항

1. **실제 사용성**: 본인이 정말로 5번 이상 사용할 만한 도구여야 합니다
2. **문제 해결**: 본인이 겪고 있는 실제 문제를 해결해야 합니다
3. **ML 모델 학습**: Assignment 5에서 최소 한 번은 ML 모델을 직접 학습시켜봐야 합니다
   - 단, 최종 결과물에서 반드시 그 모델을 사용해야 하는 것은 아닙니다
   - 학습 및 평가 후, 공개 모델이나 LLM API가 더 적합하다면 그것을 사용해도 됩니다
   - 중요한 것은 모델 학습 및 평가 경험입니다

### TL;DR - 이것이 핵심입니다

| 항목 | 설명 |
|------|------|
| **실제 문제** | 본인이 정말로 겪고 있는 문제인가? |
| **실사용** | 만든 후 5번 이상 사용할 것인가? |
| **제안서 수준** | 평소에 고민하던 내용을 1-2시간에 정리한 수준으로 충분 |

## 평가 기준

### 최종 프로젝트 평가 기준 (Assignment 6)

**이 과제는 Assignment 3, 4, 5, 6을 아우르는 전체 프로젝트의 한 부분입니다.** 최종 프로젝트는 **최종 보고서(Final Report)와 발표(Presentation)**를 기준으로 평가됩니다.

| 항목 | 배점 | 설명 |
|-----|------|------|
| **문제 정의 및 실용성** | 20% | 실제 본인의 문제를 해결하는가? 5번 이상 사용했는가? |
| **데이터 수집 및 분석** | 20% | 적절한 데이터를 수집하고 분석했는가? |
| **ML 모델 학습 및 평가** | 20% | Assignment 5에서 모델을 직접 학습하고 평가했는가? |
| **최종 보고서 및 발표** | 40% | 프로젝트 과정과 결과를 명확히 설명했는가? |

### Assignment 3 평가 기준

이 과제는 **Pass / Non-Pass**로만 평가됩니다.

**Pass 기준**:
1. **제출 완료**: PR이 정상적으로 생성되고 merge 가능한 상태
2. **내용 적절성**: 다음 항목들이 포함되어 있음
   - 해결하고자 하는 실제 문제
   - 제안하는 솔루션
   - ML 활용 계획

**Non-Pass 사유**:
- PR 형식이 맞지 않아 merge 불가능
- 제안서가 너무 간략하여 프로젝트 내용 파악 불가
- ML과 무관한 프로젝트
- 명백히 실현 불가능한 프로젝트

### 예시 1: arXiv 논문 자동 추천 Slack 봇

#### 해결하고자 하는 문제

연구실이나 스터디 그룹에서 매일 쏟아지는 논문들을 일일이 확인하기 어렵습니다. arXiv에는 매일 수백 편의 논문이 올라오지만, 내 연구 분야에 관련된 논문만 골라서 보기가 힘듭니다.

**실제 사용 시나리오**: 매일 아침 9시, 전날 올라온 NLP 관련 논문들이 자동으로 Slack에 요약과 함께 전달되어, 연구실 팀원들과 함께 흥미로운 논문을 공유하고 토론합니다. 한 달에 최소 20일 이상 사용할 예정입니다.

#### 제안하는 솔루션

arXiv API를 활용하여 특정 분야(예: cs.CL, cs.LG)의 최신 논문을 자동으로 수집하고, LLM을 활용해 3줄 요약과 함께 Slack 채널에 전송하는 봇입니다.

**주요 기능**:
- 매일 정해진 시간에 arXiv에서 새 논문 자동 수집
- LLM을 활용해 Abstract를 3줄로 요약
- 관심 있는 카테고리만 선택적으로 구독
- Slack 채널에 자동으로 메시지 전송

#### ML 활용 계획

**Assignment 4**:
- arXiv API를 통해 최근 3개월간 cs.CL 분야 논문 500편 수집
- 논문 메타데이터 분석 (제목, Abstract, 주요 키워드 등)

**Assignment 5**:
- 논문 Abstract 분류 모델 학습 (예: BERT 기반)
  - 목표: 논문의 세부 주제 자동 분류 (예: Machine Translation, QA, Summarization)
  - 평가: Accuracy, F1-score 측정
- 최종 배포 시 성능이 좋으면 학습 모델 사용, 아니면 LLM API 사용

**Assignment 6**:
- 실제 사용하며 유용성 검증

### 예시 2: 학과 공지 자동 분류 및 알람 디스코드 봇

#### 해결하고자 하는 문제

학과 홈페이지 공지사항을 매일 확인하는 것이 번거롭고, 중요한 공지(장학금, 수강신청 등)를 놓치는 경우가 많습니다. 모든 공지를 다 읽기에는 시간이 부족하고, 본인에게 관련 없는 공지도 많습니다.

**실제 사용 시나리오**: 매일 오후 6시, 새로운 공지가 있으면 자동으로 디스코드 DM으로 수신하며, 장학금 관련 공지는 즉시 알림을 받습니다.

#### 제안하는 솔루션

학과 홈페이지를 자동으로 크롤링하여 공지사항을 수집하고, ML 모델로 카테고리를 분류한 뒤, 사용자가 관심있는 분야만 디스코드로 알림을 보내는 봇입니다.

**주요 기능**:
- 매시간 학과 홈페이지 공지사항 자동 확인
- 공지를 카테고리별로 분류 (학사, 장학금, 행사, 취업 등)
- 관심있는 카테고리만 알림 수신

#### ML 활용 계획

**Assignment 4**:
- 과거 공지사항 500개 크롤링
- 카테고리 라벨링 및 텍스트 분석

**Assignment 5**:
- 텍스트 분류 모델 학습 (예: Logistic Regression, KoBERT 등)
  - 목표: 새 공지의 카테고리 자동 예측
  - 평가: Cross-validation으로 정확도 측정

**Assignment 6**:
- 실제 사용하며 성능 검증 및 개선

### 예시 3: 개인화된 영한 번역 도구

#### 해결하고자 하는 문제

기존 번역기(Google, DeepL)는 좋지만, 본인이 선호하는 표현이나 문체를 반영하지 못합니다. 특히 기술 문서나 논문 번역 시, 전문 용어의 일관성이 부족합니다.

**실제 사용 시나리오**: 논문 읽으면서 영어 문장을 즉시 번역하며, 본인이 선호하는 용어로 자동 변환됩니다. 주 5회 이상 사용할 예정입니다.

#### 제안하는 솔루션

기본 번역 엔진(DeepL API)에 개인화 레이어를 추가하여, 사용자의 피드백을 학습해 개인 맞춤형 번역을 제공하는 도구입니다.

**주요 기능**:
- DeepL API로 초벌 번역 제공
- 번역 결과를 수정하면 학습 데이터로 저장
- 사용자의 선호 표현을 학습하여 더 나은 번역 제공

#### ML 활용 계획

**Assignment 4**:
- 본인의 과거 번역 데이터 100쌍 수집
- 기존 번역기 결과 vs. 본인이 수정한 결과 분석

**Assignment 5**:
- Post-editing 모델 학습 (예: Seq2Seq, T5 등)
  - 데이터: (원문, 기계번역, 본인의 수정본) 삼중쌍
  - 목표: 기계번역 결과를 본인 스타일로 교정
  - 평가: BLEU score, 수동 평가

**Assignment 6**:
- 실제 사용하며 성능 개선 및 검증

## FAQ

**Q: ML 모델을 반드시 학습시켜야 하나요?**
A: 네, Assignment 5에서 최소 한 번은 ML 모델을 직접 학습시켜봐야 합니다. 단, 최종 결과물에서 그 모델을 사용해야 하는 것은 아닙니다. 학습 및 평가 후, 공개 모델이나 API가 더 적합하다고 판단되면 그것을 사용해도 됩니다.

**Q: 프로젝트 주제를 바꿀 수 있나요?**
A: Assignment 3 제출 마감 전까지는 자유롭게 수정 가능합니다. 이후에는 담당 교수/조교와 상의 후 변경 가능합니다.

**Q: 팀 프로젝트인가요?**
A: 아닙니다. 개인 프로젝트입니다.

**Q: 프로젝트 예시를 그대로 사용해도 되나요?**
A: 예시는 참고용입니다. 예시와 동일한 프로젝트는 권장하지 않습니다. 본인만의 문제와 아이디어를 추가하세요.

**Q: 정말로 5번 이상 사용해야 하나요?**
A: 네, 이것이 가장 중요한 기준입니다. 본인이 실제로 사용하지 않는 도구는 의미가 없습니다. 최종 보고서에서 사용 증거(로그, 스크린샷 등)를 제출해야 합니다.

**Q: 데이터를 직접 수집해야 하나요?**
A: 공개 데이터셋을 사용해도 되지만, 본인의 문제에 맞게 가공하거나 추가 수집이 필요할 수 있습니다. Assignment 4에서 데이터 수집 및 분석 과정을 거칩니다.

**Q: LLM API 사용만으로도 괜찮나요?**
A: 최종 결과물에서는 괜찮습니다. 하지만 Assignment 5에서는 최소 한 번 모델을 직접 학습시켜봐야 합니다. 예를 들어, 번역 봇을 만든다면:
- Assignment 5: 번역 품질 평가를 위해 간단한 분류 모델 학습
- Assignment 6: 최종적으로는 GPT-4 API 사용 (학습한 모델보다 성능이 좋다고 판단)
